# Рабочий процесс по плану Forecast Economy v2

## Фазы: начал — доделал

- **Фаза 0 (подготовка):** критерии закрытия и что остаётся только в кабинете Яндекса — **[docs/phase0_closeout.md](phase0_closeout.md)**.
- **Не начинать следующую фазу**, пока **текущая не закрыта** по критериям плана (вертикальный срез: что обещано в фазе — сделано и проверяемо).
- Если фаза большая — делить на **подзадачи внутри фазы**, но **не** считать фазу завершённой до полного критерия «готово» для этой фазы.

## Git и окружения

- **GitHub (`git push`)** — основной способ фиксировать прогресс; коммиты должны быть **согласованы** с тем, что реально сделано.
- **Прод-сервер (SSH / docker на 5.129.204.194 и т.п.)** — **не деплоить автоматически** и **не без явного запроса** владельца. Разработка и проверка — **локально** (и при необходимости CI), выкладка на сервер — **отдельным шагом**, когда скажете.

## История проекта

- События и решения фиксировать в `history_of_project.md` (append-only).

## Агент и проверки

- Агент **сам** поднимает то, что может: `docker compose`, `pytest`, `npm test`, проверка `/api/v1/health`, без списков «вам нужно запустить…» в конце ответа.
- Локально одной командой (эквивалент CI): **`./scripts/check-all.sh`** из корня репозитория.
- Если Docker/сеть в среде недоступны — зафиксировать в `history_of_project.md`, тесты в CI/локально в venv всё равно гонять.

## Визуальный стандарт (UI)

- Интерфейс — осознанный продуктовый уровень: типографика, отступы, палитра в тон проекта (champagne / нейтрали), состояния ошибок и фокуса, без «дефолтного» визуального мусора. Новые экраны согласовывать с `frontend/src/index.css` и существующими компонентами.

### Обязательная проверка в браузере (после правок UI)

- Любое изменение страниц/компонентов фронтенда **не считается завершённым**, пока агент **не открыл** маршрут в браузере, **не сделал snapshot**, **не прочитал консоль** (инструмент браузера: сообщения console — ошибки/warning) и **не убедился визуально** (при необходимости — скриншот).
- Если dev-сервер недоступен в среде — явно записать это в `history_of_project.md` и что проверено альтернативно (например, только unit-тесты / storybook).
