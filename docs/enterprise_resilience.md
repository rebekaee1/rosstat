# Устойчивость и «enterprise»-уровень решений

Краткий чеклист того, что закладывать при доработках API, парсеров и UI.

## Внешние источники (ЦБ, Росстат и т.д.)

- **Валидация ответа до парсинга:** минимальная длина, ожидаемые маркеры разметки, отсутствие страниц ошибки/редиректа на заглушку.
- **Таймауты и ретраи** на уровне HTTP-клиента; логирование `final_url`, размера тела, кода ответа.
- **Идемпотентность** при записи в БД (upsert по дате/ключу), без дублей при повторном fetch.
- **Деградация:** если источник недоступен — явная ошибка в логах/метаданных, а не «тихий» пустой ряд без объяснения.

## API для фронтенда

- **Пустой ряд** — валидное состояние (200 + пустой массив или явное поле `empty_reason`), не путать с 500.
- **Версионирование/кэш:** заголовки или query для инвалидации при смене логики бэка.
- **Ошибки:** стабильная форма `{ detail, code }` где возможно; фронт показывает баннер + retry.

## Фронтенд

- **Состояния:** loading, empty, error, stale (refetch); не оставлять «пустой экран» без подсказки.
- **Пустой график/таблица:** явный empty state с текстом «почему» (нет данных на бэке, фильтр, ошибка сети).
- **Retry** там, где пользователь может восстановить ситуацию без перезагрузки страницы.
