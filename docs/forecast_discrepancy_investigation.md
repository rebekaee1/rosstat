# Расхождения прогнозного файла — расследование

**Источник:** правки `v3/edit_001` и `v3/edit_002` (видео НА правки 3).

## Заявленная проблема

Пользователь указывает: при сравнении прогнозного файла (Excel/CSV) с тем, что отображается на сайте, начиная **с августа** наблюдается расхождение **порядка 0.5 п.п.** (процентных пунктов) для прогноза ИПЦ. До августа значения совпадают, далее — расходятся.

## Запрос пользователю

Прислать:
1. **Прогнозный файл** (xlsx/csv) — желательно полный, с пометкой «Эталон».
2. **Скриншот раздела сайта**, где видны расходящиеся значения, с открытой DevTools-консолью (Network → response of `/api/v1/indicators/cpi/inflation`).
3. **Дату snapshot**: когда файл был сгенерирован (timestamp).
4. **Описание модели**: какая модель ожидалась согласно файлу.

## Гипотезы (до получения файла)

Возможные причины расхождения 0.5 п.п. начиная с августа:
1. **Lag в тренировочных данных.** Backend модель тренировалась на данных, доступных на N дату, а файл — на N+M дату. После N+M вышел релиз, который переучил модель.
2. **Переход с фактов на прогноз.** Если в файле август = факт, а на сайте август = прогноз (или наоборот), смещение 0.5 п.п. ожидаемо.
3. **Корреляция с переходом ETL.** Дата `12 апреля` была ETL-катастрофой (см. history_of_project.md, 2026-04-12). После рестарта ETL могли измениться какие-то даты.
4. **Сезонные коэффициенты.** Если в файле модель использует Census X-13, а на сайте — Holt-Winters, август (резкий рост цен на ЖКХ) — типичная точка расхождения.

## План расследования (после получения файла)

1. Сделать diff: значения файла vs `/api/v1/indicators/cpi/forecast` за тот же период.
2. Проверить дату последнего фактического ИПЦ в БД vs дата файла.
3. Запустить `forecaster.py::train_inflation_12m` локально с фиксированным `as_of_date` = дата файла, сравнить.
4. Если расхождение реально — оформить как баг в `app/services/forecaster.py` или согласовать с источником эталонного файла.
